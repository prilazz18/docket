System Architecture & Implementation Defense

DocketPro - Regional Trial Court (RTC) Sipalay City

Executive Summary:
This document outlines the technical methodologies, security protocols, structural integrations, and financial valuations of the DocketPro web application. It serves to justify the system's robustness, scalability, and military-grade cloud architecture for public deployment within the Philippine justice sector.

1. Core Technology Stack

The application was intentionally built as a Custom Single Page Application (SPA) to ensure lightning-fast navigational rendering typical of enterprise-grade local software. By completely omitting bloated legacy frameworks (like WordPress or jQuery), the system executes raw JavaScript ES Modules natively in the browser.

  • Frontend Interface: Vanilla JavaScript (ESM Core), HTML5, Custom Built CSS3 (Glassmorphism UX Design).
  • Backend Integrations: Google Firebase SDK (Version 10.9.0 via Content Delivery Network).
  • Authentication: Firebase Secure signInWithEmailAndPassword Identity Toolkit.
  • Database Infrastructure: Firebase Cloud Firestore (NoSQL Scalable Document Storage).

2. Advanced Data Logic & Routing

We developed a powerful client-side computational engine to radically lower server costs by shifting heavy computing (filtering/sorting) directly into the user's browser processor.

  • 5-Dimensional Advanced Filter Matrix: A custom logic array iterates over the main dataset inside applyFilters(), allowing users to cross-reference constraints recursively (Place ∩ Case Type ∩ Status ∩ Filing Month ∩ Filing Year).
  • Separation of Concerns (Events vs Cases): Non-litigation calendar items (meetings, holidays) bypass the central statistical engine using an explicit logic flag (isEvent: true). This protects standard metrics calculations (e.g. "Total Active Cases") from data pollution while injecting both items simultaneously to the Calendar view and the 7-Day Dashboard Radar.
  • Asynchronous Live Syncing (onSnapshot): The application acts as a permanent live radar. Instead of aggressively polling the server (which rapidly consumes budget limits), the app subscribes directly to Firestore. The millisecond another user saves or deletes a record remotely, Firestore "pushes" the byte update to onSnapshot, forcefully redrawing the UI across all devices.

3. User Interface (UI) Customizations

The aesthetic was designed specifically to reflect premium, futuristic institutional design while maximizing accessibility for aging administrative personnel.

  • Dynamic CSS Variable State Toggle: A LocalStorage-tied ☀️/🌙 switch injects a .light-mode physical override up to the DOM roots. This instantly dynamically recalculates every var(--primary) hexadecimal hex code, switching the neon deep blacks to Apple-esque slates and corporate blues, accommodating visual fatigue efficiently.
  • Elimination of Legacy Prompts: Standard ugly browser alerts and prompts completely stall JavaScript threads. These were physically replaced with beautifully layered DOM Modals utilizing animated @keyframes shake CSS overrides built to mimic premium AppStore security layers.
  • Automated Philippine Time (PHT): Using `toLocaleTimeString` forced statically into the `Asia/Manila` time-zone ensures the bottom right Dashboard clock ticks synchronously across any device (even if foreign tourists/auditors access the app with misconfigured laptop time-zones).

4. Dual-Layer Military Grade Security

The core objective of securing confidential court docket files was accomplished utilizing a two-pronged redundancy architecture.

Layer 1: Frontend User interface Control (RBAC)

While secondary "Viewer" personnel are authorized via Firebase Authentication to log in and look at the dashboards, explicit Action Buttons (New Case, Edit, Delete) fire an impenetrable Security Checkpoint Modal. If personnel lack the manual fallback password, the UI strictly blocks them from interfacing with the submission forms entirely.

Layer 2: Backend Cloud Firestore Operations Rules

If Layer 1 is somehow hypothetically bypassed using manual malicious injections, the secondary backend security protocol automatically shuts off the attack locally at the firewall level inside the primary Google Server. The following strict ruleset was permanently deployed to Firestore:


rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /cases/{caseId} {
      allow read: if request.auth != null;
      allow write: if request.auth != null && request.auth.uid == 'ADMIN_UID_STRING';
    }
  }
}

This asserts that nobody on Earth can trigger a Write (addDoc, updateDoc, deleteDoc) packet operation aside from the unique encrypted UID of the Supreme Administrator.
